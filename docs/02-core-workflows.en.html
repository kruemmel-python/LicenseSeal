Core Workflows: Inject, Audit, Compare, Remove

Overview

The four most important basic commands are:

inject  → Marker einfügen oder aktualisieren
audit   → Projekt prüfen
compare → zwei Projekte strukturell vergleichen
remove  → Marker entfernen

Inject

Standard

licenseseal inject . --license MIT --owner "ACME GmbH"

With Project Name

licenseseal inject . --license MIT --owner "ACME GmbH" --project "payment-service"

Update existing markers

licenseseal inject . --license MIT --owner "ACME GmbH" --update

Include configuration files (Default: source code files are prioritized. If you also want to mark configuration files):

By default, source code files are prioritized. If you also want to highlight configuration files:

licenseseal inject . --license MIT --owner "ACME GmbH" --include-configs

Exclude folder

licenseseal inject . --exclude-dir node_modules --exclude-dir dist --exclude-dir build

Signed Markers

For stronger integrity, you can generate keys:

licenseseal keygen private.pem public.pem

After signing and injecting:

licenseseal inject . --license MIT --owner "ACME GmbH" --sign-key private.pem

Audit with signature verification:

licenseseal audit . --verify-key public.pem

Audit

licenseseal audit .

With GitHub annotations for CI:

licenseseal audit . --format github

With signature verification:

licenseseal audit . --verify-key public.pem

Compare

Compare an original project with a suspicious project:

licenseseal compare C:\repos\original C:\repos\suspected --output compare.json

Typical output includes:

  • Structural similarity
  • License status
  • affected files
  • Warnings about missing markers

Remove: With backup:

licenseseal remove .

With Backup:

licenseseal remove . --backup

Write Policies

Depending on the project status, a write policy can be useful. Examples:

licenseseal inject . --write-policy skip-existing
licenseseal inject . --write-policy overwrite

Use licenseseal inject --help to see available options in the current build.

Best Practices

  • Use --dry-run before modifying a large repository.
  • Close generated folders.
  • Use signatures for releases.
  • Use CI to prevent markers from accidentally disappearing.
  • Document Owner and license consistently.

Mini-workflow for real projects

git checkout -b chore/licenseseal
licenseseal inject . --license MIT --owner "ACME GmbH" --project "core-api" --backup
licenseseal audit .
git diff
git add .
git commit -m "Add LicenseSeal provenance markers"

Exercise

  • Mark a test project
  • Edit a marked file
  • Run an audit with signature verification if you use signed markers
  • Compare two slightly different project copies using compare
