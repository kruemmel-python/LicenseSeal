06. Registry, Firehose and OSINT

Objective

This chapter explains how LicenseSeal evolves from local audit to continuous IP radar.

Registry

The Registry stores:

  • Projects
  • Fingerprints
  • Honey-Logic Signatures
  • Scan Results
  • Evidence Items
  • Scores
  • Time Points

Initialize:

licenseseal registry init

Register Project:

licenseseal registry register .

Search or list:

licenseseal registry list
licenseseal registry search "payment-service"

Firehose

The Firehose scans candidates:

licenseseal firehose scan C:\repos\suspected

Enter a Git URL if Git is available:

licenseseal firehose scan https://github.com/example/suspected-repo.git

Queue Mode

For enterprise scaling, you can place Firehose jobs in a queue:

licenseseal firehose enqueue https://github.com/example/suspected-repo.git
licenseseal firehose worker

Redis/Celery are optional. Without infrastructure, the local mode remains usable.

OSINT

OSINT modules actively search for candidates on platforms like GitHub or GitLab.

licenseseal osint --help

Typical idea:

licenseseal osint github --query "_ls_fold_a1b2" --enqueue

Depending on the build, specific subcommands may vary. Check:

licenseseal osint --help

Recommended Enterprise Workflow

1. Eigenes Projekt markieren
2. Projekt in Registry registrieren
3. OSINT sucht Kandidaten
4. Kandidaten landen in Queue
5. Worker scannen Kandidaten
6. Evidence Items werden gespeichert
7. Control Plane zeigt Alerts
8. Legal-Team prüft Reports

Privacy and Rate Limits

At public APIs, the following applies:

  • Securely Store API Token
  • Rate Limits to consider
  • do not clone unnecessary data
  • only scan legally accessible sources
  • review results internally before taking action

Exercise

  • register a test project
  • copy it into a second folder
  • Scan the second folder with Firehose
  • Check which Evidence Signals are generated
